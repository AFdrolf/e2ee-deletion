\section{Message Deletion in the Wild}\label{sec:attacks}
Having introduced our threat model, we now turn to the motivating question behind this work: do real-world applications safely delete messages? We explore this question through the lens of two case-study applications, WhatsApp and Signal. We reverse engineered implementation details of their message deletion features, and performed a security analysis in the context of our threat model. Our analysis reveals an unsatisfactory landscape: we show numerous vulnerabilities in both applications (six in WhatsApp, and two in Signal), allowing for recovery of deleted messages across all deletion features.

\paragraph{Methodology and experimental setup} We performed our analysis of both applications using Android devices, with the latest version at the time of each application installed.\footnote{Version 2.26.11.74 for WhatsApp, and \andres{TODO} for Signal.} (As we discuss later on, however, we believe our findings are general, and not Android-specific.) Similar to prior works on attacks against messaging applications~\cite{fabrega2024injection}, we used emulated devices~\cite{android-emulator-1} for ease of experimentation, allowing us to easily reset the phones if needed and test under numerous setups and conditions. This was sufficient for our purposes, since emulated devices copy both the software \emph{and} hardware of physical devices~\cite{android-emulator-2}, and thus application files (which are the key component of our threat model) are identical as on real devices; indeed, the main use case of emulated devices is for testing applications during development, to be later run on physical end-user devices~\cite{android-emulator-1}.

Recall that deletion security considers an adversary with full, physical access to a device, which allows them to obtain the local application state. In general, access to a device is sufficient for access to its local files (e.g., by performing a full memory dump or device backup). For specific device types, however, there are often easier ways to navigate their file system. For example, in Android, a simple solution is to enable root access~\cite{sun2015android} on the device, which gives unfettered access to its file system. We opt for this approach in our experiments, but note that our forthcoming security analyses are agnostic to the method used to inspect the device's content. In particular, root access allows us to use Android Debug Bridge~\cite{adb} to explore the device's file system via a command-line interface.

\paragraph{Architecture overview.} Before presenting our vulnerabilities, we first provide necessary context by summarizing the structure of WhatsApp's application state. We provide a general overview here, and fill in additional details as needed during our vulnerability descriptions that follow. 

As reported by prior work~\cite{fabrega2024exploiting}, WhatsApp stores information about sent and received messages in a single SQLite database, which is serialized using SQLite's standard database file format~\cite{sqlite-database-file-format}. The schema for this database is complex, involving (at the time of writing) 285 tables. For our purposes, the most relevant is the ``$\messagesTable$'' table, where each row corresponds to a single (sent or received) message, storing its text content, timestamp, conversation it is part of, and other metadata. This table acts as a central hub, being referenced by numerous other tables that store additional message metadata. (We discuss details of $\messagesTable$ and other relevant tables later in the section.) For convenience, moving forward we use shorthand $\database$ to refer to this database.

For messages that contain media (e.g., images or videos), WhatsApp stores these files as separate binaries in a dedicated BLOB store. Metadata about media files is then stored in the $\mediaTable$ table in $\database$, where each row stores information about a single file, including the primary key in $\messagesTable$ for its corresponding message, the path to the file's location in storage, its size, etc. Similar to the $\messagesTable$ table, $\mediaTable$ acts as the central hub for media information, and is referenced by auxiliary tables that store more fine-grained media metadata.

In addition to the above, the application state contains numerous other files. However, these are not relevant to our purposes, so we omit description of them for brevity.

We reversed-engineered how this application state (and, in particular database $\database$) is updated as a result of WhatsApp's various message deletion features, following the general methodology mentioned at the start of the section. We now proceed to explain our findings, ordered by the type of content they target (and, thus, enable recovery of): text, media, and other.

\paragraph{Vulnerability \#1: text recovery.} We start with our most general finding, where we show how an adversary can recover the text of any deleted (sent or received) message, irrespective of the message deletion feature that was used. We depict an example setting in Figure~\ref{}. At a high level, this vulnerability stems the way in which deletion is implemented in WhatsApp's \emph{search index}. As we will see, this allows recovery of deleted messages by inspecting the serialized index. This security issue is fairly technical, relying on many low-level details of WhatsApp's index and how it is encoded in database $\database$. So, due to space constraints, we highlight the main ideas here, and defer full details to Appendix~\ref{}.

\smallskip\textbf{\emph{WhatsApp's search index.}} 
To support efficient search, WhatsApp uses SQLite's FTS4 module~\cite{sqlite-fts4} to implement full-text search on all sent and received messages. An FTS4 index is logically structured as an \emph{inverted} index, mapping \emph{terms} to the \emph{documents} (and their positions therein) in which they appear. In WhatsApp, terms (i.e., index entries) roughly correspond to individual words,\footnote{More precisely, in WhatsApp, the terms in message $m$ are the words that result after casting into $m$ all lowercase, and replacing all non-alphanumeric characters with whitespace.} and documents to the messages in which they are contained; messages are represented by a \emph{message identifier}, which are assigned sequentially (and are also the primary keys in the $\messagesTable$ table). Thus, the index has an individual entry for every word across all messages, mapping to a list of all messages (and their position within the message) in which the word is present. For example, if a user has (in order) messages ``Hi Alice'' and ``Hi Bob'', their index would map ``hi'' to $[(1, 1), (2, 1)]$, ``alice'' to $[(1, 2)]$, and ``bob'' to $[(2, 2)]$.

Like all other message information, the index is stored in database $\database$. The way in which an FTS4 index is encoded and arranged within a database is complex; for clarity, we provide a simplified treatment here, and refer readers to 
Appendix~\ref{} and~\cite{sqlite-fts4} for details.
At a high-level, an FTS4 index is organized as a collection of multiple B-trees, with term-document mappings (i.e., word-message mappings in WhatsApp) stored in the leaves. Concretely, a leaf contains a sequence of term and \emph{document list} (doclist) pairs, where a doclist consists of document-position pairs (e.g., $[(1, 1), (2, 1)]$ in our example above). Term-doclist pairs within a leaf are ordered lexicographically by term, and document-position pairs within a doclist are ordered by document id.
See Figure~\ref{} for an example. 

The B-trees are organized into levels. Whenever a message is added to the index, a new level-0 B-tree is created, containing only the terms in this message, each mapping to the message's identifier (i.e., its primary key in $\messagesTable$) and its position in it. That is, let $m = (w_1, \dots, w_q)$ be the $n$-th (sent or received) message, where $w_i$ are its terms, in order. This results in a level-0 B-tree containing (only) pairs $(w_1, [(n, 1)]), (w_2, [(n, 2)]), \dots, (w_q, [(n, q)])$, arranged lexicographically by $w_i$.\footnote{If there are two $w_i$ and $w_j$ such that $w_i = w_j$ and $i < j$, then there is only a single pair for them of the form $(w_i, [(n, i), [n, j]])$.} After a certain number of level-0 B-trees are present (16, in the case of WhatsApp), they are merged into a level-1 B-tree, containing the term-doclist pairs of all 16 level-0 B-trees, which are then deleted. In particular, if a term is present in multiple level-0 B-trees, it is stored as single entry in the level-1 B-tree, with all document-position pairs in the term's (single) doclist.\andres{Previous sentence candidate to remove.} This progression continues: in general, after 16 level $i$ B-trees are created, they are replaced by a new level $i+1$ B-tree. So, as the index grows, it consists of multiple B-trees of different levels and sizes, which are traversed during a search query. This means that a term, while unique within a B-tree, may be repeated across different trees, which get consolidated when/if these trees merge.\andres{Previous sentence candidate to remove.}

The nodes of all B-trees are stored in the $\messagesIndexSegdirTable$ and $\messagesIndexSegmentsTable$ tables of $\database$---root notes in the former, and all other nodes in the latter. Thus, scanning these tables allows inspecting the physical encoding of the index. The maximum size of a node is the database's page size (4096 bytes in WhatsApp), and so lower-level B-trees often require just a single root node.

\smallskip\textbf{\emph{Text deletion.}}
Application-level deletions trigger a cascade of changes in $\database$. These changes are generally consistent across all message deletion features. First, the message's text is removed from the $\messagesTable$ table. For silent features (property \#3 in our systematization) such as local deletion, the row is deleted entirely; and for visible features such as unsending messages, the row is kept, but the cell storing its text content is emptied, replaced by a NULL value. Notice that, for silent features, the $\messagesTable$ table thus leaks information: row deletions result in a \emph{skip} in primary keys, revealing that \emph{some} message was deleted (but not which or in which chat). That is, if the $i$-th message is deleted, the table will have a row with primary key $i-1$ followed by one with $i+1$.

While identifying that deletion occurred can be sensitive in many contexts, our focus is on message recovery. In addition to the $\messagesTable$ table, the second prominent change is that deletion features trigger a deletion request to the index. However, deletions in FTS4 are only \emph{logical}: instead of \emph{erasing} content, deletions create new \emph{tombstone entries}. Concretely, deleting the $n$-th message $m = (w_1, \dots, w_q)$ creates a \emph{new} index entry containing $(w_1, [(n, 0)]), (w_2, [(n, 0)]), \dots, (w_q, [(n, 0)])$. As usual, this entry is stored as a new level-0 B-tree, stored in $\messagesIndexSegdirTable$ (and $\messagesIndexSegmentsTable$, if the message is long and does not fit in a single root node). Such an entry encodes that terms $w_i$ appear in ``position 0'' of message $n$, which instructs future search queries to omit $m$ from the search results for any $w_i$.

Deletions in the search index are thus only at the application-level: both the original index entry (when the message was sent or received) and the new tombstone entry are still present in the physical index in storage, and can thus be recovered via direct inspection of the B-tree nodes in the $\messagesIndexSegdirTable$ and $\messagesIndexSegmentsTable$ tables.

\paragraph{Vulnerability \#1: write-ahead logging.} The first security issue stems from the \emph{journal mode} used for the SQLite database $\database$. In general, a database's journal mode is the mechanism by which new transactions are committed to the database.
